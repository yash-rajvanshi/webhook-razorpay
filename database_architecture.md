# 🗃️ MongoDB Architecture & Data Flow

This document details exactly how the Vercel server interacts with MongoDB Atlas, tracking user premium subscription events robustly.

---

## 1. Connection Strategy (The Serverless Problem)
In a traditional Node.js server, you establish a database connection once when the server boots. However, Vercel Serverless Functions spin up and destroy themselves rapidly based on traffic. If your webhook connects to MongoDB every single time a request comes in, you will instantly exhaust your MongoDB connection pools and crash the database.

**The Solution (`api/_lib/db.js`)**: 
We implemented a strict **Connection Cashing** protocol using `clientPromise`. When the first request forces Vercel to boot up the function, it connects to Mongo and stores that connection reference globally. Any subsequent webhooks that hit that same warmed-up lambda container will completely skip the connection phase and reuse the open tunnel, dropping latency to milliseconds!

---

## 2. What We Save (The `users` Collection)
Whenever a user pays through Razorpay, the Razorpay Webhook passes the `telegram_id` (which we gracefully smuggled inside the `notes` payload originally). We extract this and immediately target the `users` collection.

Here is the exact schema mapping of what we save for every user:

```json
{
  "_id": "ObjectId('...')",                  // Automatically generated by MongoDB
  "telegram_id": "123456789",              // (String) The primary unique key linking back to Telegram API
  "username": "yash_rajvanshi",            // (String) Captured during the /start flow
  "first_name": "Yash",                    // (String) Helpful for admin readability in the raw DB
  "status": "active",                      // (String) State tag
  "lastPaymentDate": "2024-04-05T00:00:00.000Z", // (Date) The exact millisecond the Razorpay webhook fired
  "expiresAt": "2024-05-05T00:00:00.000Z",       // (Date) 30 days from the payment (Dynamic)
  "lastPaymentId": "plink_xyz987"          // (String) The Razorpay Payment Link ID to trace receipts
}
```

---

## 3. Database Write Logic (Upserts & Extensions)
Database changes **only** happen natively inside `api/razorpay-webhook.js` during the `payment_link.paid` event. 

We do not use standard "Inserts". We use intelligent **Upserts** (`upsert: true` flag):
1. **The Query**: It looks for `{ telegram_id: "123456789" }`.
2. **If NO user exists or it's EXPIRED**: It creates a fresh document, stamping `expiresAt` exactly `30 days` into the future.
3. **If user EXISTS and is ACTIVE**: It gracefully extracts their current `expiresAt` date, and strictly *adds 30 days to whatever that previous expiry date was*, effectively letting users stack subscriptions without losing overlapping days!

---

## 4. Database Read Logic
The DB is queried passively inside `/api/telegram-webhook.js`.
When a user fires the `/status` command:
- The bot extracts their `ctx.from.id`.
- The bot queries the `users` collection via `collection.findOne()`.
- It maps the `expiresAt` Date object against Node's `new Date()` (the current timestamp). If the Date is in the past, it organically rejects them. If the Date is in the future, it formats it cleanly so the user knows exactly how many days they have left!
